# Maestro Guidelines

Этот файл фиксирует правила, по которым нужно развивать и поддерживать `Maestro`-тесты в этом проекте.

## Основные источники

При работе с `Maestro` нужно в первую очередь опираться на официальные источники:

- `https://docs.maestro.dev`
- `https://maestro.dev/blog`

Если возникает вопрос по:

- синтаксису `Flow`
- `runFlow`
- `env`
- `parameters`
- `constants`
- `JavaScript`
- `output`
- `maestro.copiedText`
- `config.yaml`
- `tags`
- структуре тестового набора

нужно сначала сверяться с официальной документацией Maestro.

## Приоритет источников

Приоритет такой:

1. Официальная документация `docs.maestro.dev`
2. Официальные статьи и best practices на `maestro.dev/blog`
3. Текущая структура и соглашения этого репозитория

Если текущая реализация в проекте противоречит официальной документации, нужно:

- указать это явно
- предложить вариант ближе к официальной документации
- не менять ничего молча в спорных местах без пояснения

## Правила для изменений

При изменении `Maestro`-тестов в этом проекте нужно:

- использовать официальную терминологию Maestro
- предпочитать решения, которые соответствуют best practices из официальной документации
- при сомнениях по синтаксису проверять документацию перед изменением файлов
- явно отмечать, где решение основано на документации, а где это локальная рекомендация для проекта

## Текущая структура проекта

В этом проекте приняты такие каталоги:

- [`.maestro/test.yaml`](/Users/sergejbursov/Documents/maestro-tests/.maestro/test.yaml) — базовый smoke-сценарий
- [`.maestro/flows`](/Users/sergejbursov/Documents/maestro-tests/.maestro/flows) — самостоятельные тестовые сценарии
- [`.maestro/common`](/Users/sergejbursov/Documents/maestro-tests/.maestro/common) — переиспользуемые подфлоу

`common`-файлы не считаются самостоятельными тестами и используются только через `runFlow`.

## Подход к структуре Flow

Нужно придерживаться таких принципов:

- один `Flow` должен проверять один пользовательский сценарий
- общие шаги нужно выносить в `common`
- данные лучше отделять от механики теста
- там, где это уместно, параметры подфлоу нужно передавать через `runFlow.env`

## Подход к JavaScript в Maestro

JavaScript в `Maestro` следует использовать только там, где он действительно нужен:

- парсинг текста
- вычисления
- сравнение динамических значений
- работа с `output`
- работа с `maestro.copiedText`

Если логику можно выразить чистым YAML без потери читаемости, предпочтительнее YAML.

Если логика становится сложной:

- использовать `output` для хранения состояния
- по возможности использовать namespace, например `output.cart`, `output.auth`
- при росте сложности рассматривать вынос логики в `runScript`
